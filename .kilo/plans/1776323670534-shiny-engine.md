# Migration Plan: Remove Firebase and Add Mongoose with Custom Auth

## Overview
This project currently uses Firebase for both authentication (Google sign-in) and data storage (Firestore). The goal is to replace Firebase with:
- **Authentication**: Custom JWT-based auth with name, email, and password fields
- **Database**: MongoDB with Mongoose ODM
- **Backend**: Next.js API routes to handle auth and data operations

## Current Architecture
- Frontend: Next.js with React
- Auth: Firebase Auth (Google sign-in only)
- Database: Firestore
- Data Models: Users and Notes (3 types: dsa, qa, general)

## Target Architecture
- Frontend: Next.js with React (unchanged)
- Auth: JWT tokens via Next.js API routes
- Database: MongoDB with Mongoose
- Backend: Next.js API routes

## Prerequisites
- MongoDB instance (connection string available)
- Environment variables for database connection and JWT secret

## Implementation Steps

### Phase 1: Setup and Dependencies
1. **Install new dependencies**
   - `mongoose` - MongoDB ODM
   - `bcryptjs` - Password hashing
   - `jsonwebtoken` - JWT token handling
   - `joi` or `zod` - Input validation (already have zod)
   - `@types/bcryptjs`, `@types/jsonwebtoken` - TypeScript types

2. **Remove Firebase dependencies**
   - `firebase`
   - `firebase-tools` (dev dependency)

3. **Environment configuration**
   - Add `MONGODB_URI` to environment variables
   - Add `JWT_SECRET` to environment variables
   - Remove Firebase config files

### Phase 2: Database Models
1. **Create Mongoose models**
   - `User` model: name, email, password (hashed), createdAt, updatedAt
   - `Note` model: userId, type, title, isFavorite, tags, content, dsa, qa, createdAt, updatedAt

2. **Database connection**
   - Create `lib/mongodb.ts` for database connection
   - Update types to use Date instead of Firestore Timestamp

### Phase 3: Authentication API Routes
1. **Create auth API routes**
   - `app/api/auth/signup/route.ts` - User registration
   - `app/api/auth/login/route.ts` - User login
   - `app/api/auth/me/route.ts` - Get current user (protected)

2. **Auth utilities**
   - `lib/auth.ts` - JWT token generation/verification
   - `middleware.ts` - Route protection middleware

3. **Update auth hook**
   - Replace Firebase auth with JWT-based auth
   - Add signup functionality
   - Update login/logout logic

### Phase 4: Notes API Routes
1. **Create notes CRUD API routes**
   - `app/api/notes/route.ts` - GET (list notes), POST (create note)
   - `app/api/notes/[id]/route.ts` - GET, PUT, DELETE (single note)
   - `app/api/tags/route.ts` - GET unique tags for user

2. **Update data fetching**
   - Replace all Firestore operations with API calls
   - Implement WebSockets or Server-Sent Events for real-time updates

### Phase 5: Frontend Updates
1. **Authentication forms**
   - Create signup/login forms with name, email, password fields
   - Add auth pages: `/auth/login`, `/auth/signup`
   - Update landing page to link to login instead of Google sign-in

2. **Update all components**
   - Navbar search functionality
   - Dashboard data fetching
   - Notes pages (list, create, edit, view)
   - Sidebar/tag filtering

3. **Remove Firebase references**
   - Delete Firebase config files
   - Remove Firebase imports and usage
   - Update error handling

### Phase 6: Testing and Migration
1. **Data migration**
   - No migration needed - starting fresh with empty database

2. **Testing**
   - Test all auth flows (signup, login, logout)
   - Test all CRUD operations for notes
   - Test search and filtering functionality

3. **Cleanup**
   - Remove unused Firebase code
   - Update README with new setup instructions
   - Update package.json scripts if needed

## Key Technical Decisions

### Authentication Strategy
- Use HTTP-only cookies for JWT tokens (more secure than localStorage)
- Implement refresh tokens for better security
- Password hashing with bcryptjs
- Input validation with existing zod schemas

### API Design
- RESTful API routes following Next.js conventions
- Consistent error handling and response formats
- Pagination for notes list (if needed for performance)

### Database Schema
- Keep similar structure to Firestore but adapt for MongoDB
- Use proper indexing for search performance
- Maintain referential integrity where needed

## Potential Challenges
1. **Real-time updates**: Firestore had real-time listeners; will implement WebSockets/Server-Sent Events
2. **Search performance**: Implement proper text indexing in MongoDB
3. **Security**: Ensure proper JWT handling and password security

## Rollback Plan
- Keep Firebase config in separate branch
- Have backup of current working state
- Can revert to Firebase if issues arise

## Success Criteria
- Users can sign up with name, email, password
- Users can log in and access their notes
- All existing functionality works (CRUD notes, search, filtering)
- No Firebase dependencies remain
- Application runs with MongoDB backend</content>
